Here's a comprehensive breakdown of SqlPen's architecture and functionality:

---

## What SqlPen Does

SqlPen is a Python ETL engine with two main jobs:
1. **Load data** (CSV/Parquet/JSON/Excel/DataFrame) into SQL databases with auto-cleaning
2. **Test/validate** that data loads correctly via a diagnostic harness

---

## Core Modules

### 1. `df_tosql` — The Main ETL Pipeline
The primary entry point for loading data. It runs a sequential pipeline:

```
Source (file/URL/DataFrame)
  → Clean column names (sanitize_dataframe_columns)
  → Cast types (auto_cast)
  → Quarantine outliers (replace_outliers_with_zero_safe)
  → Auto-profile & infer PK (optional)
  → Dump schema JSON (optional)
  → Execute CRUD (insert / upsert / update / replace)
```

Modes via `if_exist`:
- `insert` — append rows, auto-creates table if missing
- `upsert` — insert-or-update by constraint columns
- `update` — update rows matching `where` conditions
- `replace` — drop + recreate table, then insert

### 2. `csv_harness` / `PipelineRunner` — The Diagnostic Harness
Runs a deterministic 3-step validation cycle against a CSV:

```
INSERT (first 60% of rows)
  → UPSERT (30% overlap + 20% new rows)
    → UPDATE (3 targeted rows)
```

After each step it runs a verification SELECT and compares results. Failures are mapped to root causes like:
- `VARCHAR overflow` → widen column
- `NOT NULL violation` → provide defaults
- `Duplicate key` → deduplicate or use upsert
- `Type casting failure` → clean source data

Outputs a `<csv_name>_harness.txt` report with full diagnostics.

### 3. `crud/` — Dialect-Native CRUD Engine
Handles the actual SQL execution for 5 dialects: PostgreSQL, MySQL/MariaDB, SQLite, Oracle, SQL Server.

Key pattern: **bulk-first with row-by-row fallback**
```
Try bulk INSERT/UPSERT
  → If fails: rollback, retry row-by-row up to tolerance limit
  → Collect per-row errors with root cause analysis
```

Returns a `CrudResult(total, success, failed, method, diagnostics)`.

### 4. `peek.py` — Database Introspection
Lightweight read-only inspection layer:

```python
import peek as pk
pk.tables("sqlite:///my.db")           # list all tables
pk.describe("users")                   # column types/nullability as DataFrame
pk.describe_full("users")              # + PKs, unique constraints, identity cols
pk.query("SELECT * FROM users")        # run SQL → DataFrame
pk.has_table("users")                  # bool check
pk.validate_upsert("users", ["id"])    # verify upsert constraints exist
```

### 5. `csvdog` — Directory Watchdog
Scans a directory for CSVs, tracks file modification times in a JSON manifest, and only re-processes changed files. Auto-infers PKs and upserts into the database.

### 6. `config.py` — Engine & Config Resolution
- `get_engine_from_env(url)` — resolves DB connection from explicit URL → `DATABASE_URL` env var → `.env` file
- `load_pipeline_config()` — loads `config.yml` with fallback defaults
- `ensure_table()` — creates/alters/drops tables from DataFrame schema with dialect-aware DDL

## Phase 1 — CLI Entry (`cli.py`)
```
sqlpen harness data.csv --table users --pk id --constraint email --clean --cast --auto-profile
  │

  └─ Call run_csv_pipeline(...)
```

---

## Phase 2 — CSV Load (`run_csv_pipeline`)
```
pd.read_csv(csv_path)
  └─ Resolve report output path → <csv_stem>_harness.txt
  └─ Instantiate PipelineRunner(engine, df, table, pk_cols, constraint_cols, ...)
```

---

## Phase 3 — Data Preparation (`PipelineRunner._prepare_data`)
Runs in strict order:
```
1. Clean   → sanitize_dataframe_columns()   remap pk/constraint names through mapping
2. Cast    → auto_cast()                    infer dates, booleans, numerics
3. Outlier → replace_outliers_with_zero_safe()   IQR-based quarantine (default threshold=0.5)
4. Profile → profile_dataframe() + get_pk()      only if --auto-profile AND pk == ['id'] or empty
```

---

## Phase 4 — Harness Slicing (`CrudTestHarness`)
The prepared DataFrame is deterministically split into 3 slices:

```
Total rows = N

INSERT slice  → rows 0      to 60%      (mutated mutable cols)
UPSERT slice  → rows 30%    to 60%  ← overlap with INSERT (mutated)
               + rows 60%   to 80%  ← brand new rows
UPDATE slice  → rows [0, 40%, 70%]  ← 3 targeted rows (mutated)
```

Mutation rules per type:
- int/float → `+1` / `+1.0`
- str → append `⟐m` marker
- bool → flipped
- datetime → `+1 day`

---

## Phase 5 — Execution Cycle (`PipelineRunner.run`)

```
Schema Fingerprint BEFORE
        │
        ▼
┌─── STEP 1: INSERT ──────────────────────────────────┐
│  auto_insert(engine, insert_df)                      │
│    → bulk attempt first                              │
│    → row-by-row fallback on failure                  │
│  Verify: SELECT WHERE pk IN (insert rows)            │
│  Validate: assert DB rows == expected insert_df      │
└─────────────────────────────────────────────────────┘
        │
        ▼
┌─── STEP 2: UPSERT ──────────────────────────────────┐
│  auto_upsert(engine, upsert_df, constrain=constraint)│
│    → overlap rows get UPDATED                        │
│    → new rows get INSERTED                           │
│  Verify: SELECT WHERE pk IN (upsert rows)            │
│  Validate: assert DB rows == expected after upsert   │
└─────────────────────────────────────────────────────┘
        │
        ▼
┌─── STEP 3: UPDATE ──────────────────────────────────┐
│  auto_update(engine, update_df, where=constraint)    │
│    → 3 targeted rows updated by constraint match     │
│  Verify: SELECT WHERE constraint IN (update rows)    │
│  Validate: assert DB rows == expected full cycle     │
└─────────────────────────────────────────────────────┘
        │
        ▼
Schema Fingerprint AFTER → diff_fingerprints() → detect schema drift
        │
        ▼
Write <csv_stem>_harness.txt report
```

---

## Phase 6 — Report Output
The `.txt` report contains in order:
1. Data slicing summary (row counts per slice)
2. Pipeline mode (clean/cast/outlier/profiler ON/OFF)
3. Schema fingerprint before (column types + hash)
4. Per step: status, rows sent, success/failed, elapsed, DML errors, verification SQL, root cause analysis
5. Schema fingerprint after + drift summary
6. Final verdict: `ALL STEPS PASSED` or `FAILURES DETECTED`

## The Core Problem

The `misc/` modules have **two conflicting engine patterns**:

| Module | Pattern | Issue |
|--------|---------|-------|
| `schema_inspector.py` | Stateless — takes `engine` per call | ✅ Clean |
| `schema_analyzer.py` | Stateful — stores `engine` in `__init__` | ⚠️ Inspector cached at construction time, stale after schema changes |
| `schema_manager.py` | Stateful — stores `engine` + caches `_inspector` | ⚠️ `refresh()` must be called manually after DDL |
| `schema_corrector.py` | Stateful — stores `conn` in `__init__` | ⚠️ Accepts both `Engine` and `Connection`, inconsistent |
| `df_align_to_sql.py` | Stateless — takes `engine` per call | ✅ Clean |
| `engine_manager.py` | Pooled registry — URL-keyed singleton engines | ✅ Correct pattern but not wired into `peek.py` |

`peek.py` wraps all of these behind a **stateless URL-based API** — but internally it creates a new engine on every call via `get_engine_from_env(url)`. This means:

1. `SchemaManager` gets a fresh engine each time `get_manager()` is called — its cached `_inspector` is always fresh, but the manager object is not reused
2. `SchemaAnalyzer` re-inspects on every `analyze()` call — correct but wasteful
3. `schema_corrector.SchemaAligner` stores the engine at `align()` call time — fine for single use but not for repeated calls

The fix is to wire `engine_manager.EngineManager._Registry` into `peek.py`'s `get_engine()` so the same engine is reused for the same URL across all calls, rather than creating a new one every time.

## Refactor Summary

### Problem identified
`peek.py` was calling `get_engine_from_env(url)` on every function call, creating a **new SQLAlchemy engine** (and a new connection pool) each time. With 15+ functions all doing this independently, repeated calls like `pk.tables()` → `pk.describe()` → `pk.query()` would spin up 3 separate pools for the same URL.

### Changes made

**`peek.py` — engine pooling via `_Registry`**
- Imports `_Registry` from `engine_manager.py` — the same thread-safe URL-keyed singleton pool already used by `EngineManager`
- `get_engine(url)` now resolves the URL string first (for env var / `.env` fallback), then hands it to `_Registry.get()` with `pool_pre_ping=True`
- Result: the same URL always returns the same engine instance across all peek calls — no redundant pool creation

**`misc/__init__.py` — package API exposed**
